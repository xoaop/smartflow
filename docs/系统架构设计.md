# 系统架构设计文档

## 一、整体架构设计
### 1.1 分层架构
本系统采用经典的分层架构设计，各层职责清晰，便于维护和扩展：

```
┌─────────────────────────────────────────────────────────────┐
│                     CLI 交互层 (OpenClaw)                    │
│  命令行解析、用户交互、指令调度                               │
├─────────────────────────────────────────────────────────────┤
│                     业务服务层                                │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐  │
│  │ 配置管理模块 │ 数据采集模块 │ 内容生成模块 │ 推送分发模块 │  │
│  └─────────────┴─────────────┴─────────────┴─────────────┘  │
│  业务逻辑实现，各模块高内聚低耦合                             │
├─────────────────────────────────────────────────────────────┤
│                     基础服务层                                │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐  │
│  │ 定时任务服务 │ 大模型服务   │ 数据存储服务 │ 日志监控服务 │  │
│  └─────────────┴─────────────┴─────────────┴─────────────┘  │
│  通用能力抽象，被业务层复用                                   │
├─────────────────────────────────────────────────────────────┤
│                     外部集成层                                │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐  │
│  │ 飞书OpenAPI │ Claude API  │ 本地文件系统 │  系统调度   │  │
│  └─────────────┴─────────────┴─────────────┴─────────────┘  │
│  与外部系统的交互适配                                         │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 架构特点
- **模块化设计**：每个业务模块独立实现，接口清晰，便于单独测试和替换
- **配置驱动**：所有业务逻辑通过配置控制，无需修改代码即可适配不同团队需求
- **多租户隔离**：多团队数据完全隔离，每个团队使用独立的飞书应用权限
- **容错性强**：各模块错误隔离，单个数据源失败不影响整体流程
- **扩展性好**：预留了扩展接口，便于后续接入更多数据源和推送渠道

## 二、核心模块设计

### 2.1 配置管理模块
**职责**：统一管理全局配置和多团队配置，提供配置的加载、校验、持久化能力。

**核心实现**：
- 配置文件采用YAML格式，易于阅读和修改
- 使用Zod进行配置Schema校验，确保配置合法性
- 支持多团队配置独立存储，数据完全隔离
- 提供配置模板生成功能，降低使用门槛

**关键文件**：
- `src/modules/config/config.schema.ts` - 配置校验规则定义
- `src/modules/config/team-config.service.ts` - 配置管理服务实现

### 2.2 数据采集模块
**职责**：从飞书多源采集结构化数据，为内容生成提供原始素材。

**核心实现**：
- 采用采集器接口设计，便于后续扩展更多数据源
- 支持文档、任务、会议三类核心数据的并行采集
- 内置数据清洗和过滤规则，自动排除无效和敏感数据
- 实现API重试和限流机制，应对飞书接口限制

**关键文件**：
- `src/modules/collector/collector.interface.ts` - 采集器接口定义
- `src/modules/collector/doc-collector.ts` - 文档采集实现
- `src/modules/collector/task-collector.ts` - 任务采集实现
- `src/modules/collector/meeting-collector.ts` - 会议采集实现
- `src/modules/collector/feishu-collector.service.ts` - 采集调度服务

### 2.3 内容生成模块
**职责**：基于采集到的原始数据，通过大模型生成结构化的周报内容。

**核心实现**：
- 提示词工程优化，确保输出格式规范、内容准确
- 大模型返回结果结构化解析，支持JSON格式输出
- 幻觉校验机制，所有生成内容必须有原始数据支撑
- 支持多级别详细度配置，满足不同场景需求

**关键文件**：
- `src/modules/generator/prompt-templates.ts` - 提示词模板定义
- `src/modules/generator/report-generator.service.ts` - 报告生成服务实现
- `src/common/llm/claude-client.ts` - Claude API封装

### 2.4 推送分发模块
**职责**：将生成的周报以卡片形式推送到飞书指定渠道。

**核心实现**：
- 飞书卡片模板化构建，支持丰富的交互效果
- 多渠道推送能力，支持群聊和私聊
- 审核流程支持，重要内容可配置人工审核
- 推送结果记录，便于后续统计分析

**关键文件**：
- `src/modules/push/card-builder.ts` - 飞书卡片构建器
- `src/modules/push/feishu-push.service.ts` - 推送服务实现

### 2.5 定时任务模块
**职责**：实现定时自动生成和推送周报的能力。

**核心实现**：
- 基于cron表达式的灵活任务调度
- 任务状态持久化，服务重启自动恢复
- 执行日志完整记录，便于问题排查
- 失败自动重试和告警机制，提高系统可靠性

**关键文件**：
- `src/modules/schedule/scheduler.service.ts` - 定时任务调度服务
- `src/common/db/sqlite.ts` - SQLite数据库封装

## 三、数据流设计
### 3.1 完整数据流
```
用户配置 → 定时触发/手动触发 → 数据采集 → 数据清洗 → 大模型生成 → 内容校验 → 卡片构建 → 推送分发 → 结果存储
```

### 3.2 数据流向说明
1. **触发阶段**：系统支持两种触发方式：
   - 定时触发：根据配置的cron表达式自动触发
   - 手动触发：用户通过CLI命令手动执行

2. **采集阶段**：
   - 根据团队配置并行拉取飞书文档、任务、会议数据
   - 对原始数据进行结构化处理，提取关键字段
   - 过滤无效数据和符合排除规则的数据

3. **生成阶段**：
   - 将结构化数据按照提示词模板格式化
   - 调用Claude大模型生成周报内容
   - 对生成结果进行校验，确保内容真实有效

4. **推送阶段**：
   - 将结构化的周报内容转换为飞书卡片格式
   - 根据配置推送到指定的群聊或用户
   - 记录推送结果和执行日志

5. **存储阶段**：
   - 保存原始采集数据、生成的周报内容、推送记录
   - 存储执行日志和统计数据，用于后续效果分析

## 四、关键技术选型
| 技术/组件 | 选型说明 | 优势 |
|---------|---------|------|
| TypeScript | 开发语言 | 类型安全，IDE支持好，便于大型项目维护 |
| Node.js | 运行时 | 异步IO性能好，适合IO密集型任务，CLI开发成熟 |
| Commander.js | CLI框架 | 成熟稳定的命令行解析库，生态完善 |
| 飞书官方SDK | 飞书对接 | 官方维护，API覆盖全面，兼容性好 |
| Anthropic SDK | 大模型对接 | 官方SDK，支持最新Claude模型，功能完善 |
| node-schedule | 定时任务 | 轻量级，支持cron表达式，使用简单 |
| SQLite | 数据存储 | 轻量级，无需单独部署，适合小体量应用 |
| Zod | 参数校验 | 类型安全的校验库，配置校验友好 |
| Winston | 日志框架 | 功能完善，支持多输出渠道，性能优秀 |

## 五、非功能设计
### 5.1 可靠性设计
- 所有外部API调用都实现指数退避重试机制
- 模块级错误隔离，单个模块失败不影响整体流程
- 任务执行状态持久化，崩溃重启后可恢复
- 关键操作完整日志记录，便于问题追溯

### 5.2 安全性设计
- 所有配置文件存储在用户目录下，权限控制严格
- 飞书API密钥和Claude API Key本地存储，不上传到任何第三方
- 多团队数据完全隔离，不会出现跨团队数据泄露
- 支持敏感词过滤，避免敏感信息进入周报

### 5.3 扩展性设计
- 各模块基于接口实现，便于替换具体实现
- 采集器接口设计，可快速接入新的数据源（如Git、Jira等）
- 推送渠道抽象，可扩展支持企业微信、邮件等推送方式
- 大模型服务抽象，可快速切换不同的大模型供应商

### 5.4 性能设计
- 多数据源并行采集，缩短整体执行时间
- 大模型提示词优化，减少Token消耗和响应时间
- 轻量级架构，资源占用低，可在低配服务器上运行
- 支持批量处理，满足多团队同时使用需求
