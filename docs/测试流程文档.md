# 测试流程文档

## 一、测试环境搭建
### 1.1 测试环境要求
- 独立的飞书测试企业，避免影响生产数据
- 测试用的飞书应用，配置和生产一致的权限
- 测试专用的Claude API Key，避免消耗生产额度
- 测试数据：包含测试文档、测试任务、测试会议

### 1.2 环境初始化
```bash
# 克隆代码
git clone <仓库地址>
cd smartflow

# 安装依赖
npm install

# 编译
npm run build

# 链接到全局
npm link

# 创建测试配置目录
mkdir -p ~/.smartflow-test/config/teams
cp config/config.example.yaml ~/.smartflow-test/config/config.yaml
cp config/team-config.example.yaml ~/.smartflow-test/config/teams/test-team.yaml

# 编辑测试配置，填写测试环境的相关信息
```

### 1.3 环境变量配置
```bash
# 配置测试环境的根目录
export SMARTFLOW_CONFIG_ROOT=~/.smartflow-test
```

## 二、测试分层策略
### 2.1 单元测试
**测试范围**：单个函数、类、模块的功能正确性
**覆盖率要求**：核心模块覆盖率≥80%
**工具**：Jest + ts-jest

#### 单元测试用例示例
```typescript
// src/modules/config/__tests__/team-config.service.test.ts
import { TeamConfigService } from '../team-config.service';

describe('TeamConfigService', () => {
  let service: TeamConfigService;

  beforeEach(() => {
    service = TeamConfigService.getInstance();
  });

  test('should create team config template correctly', () => {
    const template = service.createTeamConfigTemplate('test', '测试团队');
    expect(template.teamId).toBe('test');
    expect(template.teamName).toBe('测试团队');
    expect(template.generate.cycle).toBe('weekly');
  });

  test('should validate config correctly', async () => {
    const invalidConfig = { teamId: '', teamName: '' };
    await expect(service.saveTeamConfig(invalidConfig as any)).rejects.toThrow();
  });
});
```

#### 运行单元测试
```bash
# 运行所有单元测试
npm test

# 运行指定模块的测试
npm test -- src/modules/config/__tests__/

# 生成覆盖率报告
npm test -- --coverage
```

### 2.2 集成测试
**测试范围**：多个模块之间的交互、与外部系统的集成
**测试内容**：
- 飞书API集成测试
- Claude API集成测试
- 数据库集成测试
- 端到端流程测试

#### 飞书API集成测试
```typescript
// src/common/feishu/__tests__/client.integration.test.ts
import { FeishuClient } from '../client';
import { TeamConfig } from '../../../types';

describe('FeishuClient Integration Test', () => {
  let client: FeishuClient;

  beforeAll(() => {
    const testConfig: TeamConfig = {
      teamId: 'test',
      teamName: '测试',
      feishu: {
        appId: process.env.TEST_FEISHU_APP_ID!,
        appSecret: process.env.TEST_FEISHU_APP_SECRET!,
        scopes: [],
      },
      // 其他配置...
    };
    client = new FeishuClient(testConfig);
  });

  test('should get tenant access token correctly', async () => {
    const token = await client.getTenantAccessToken();
    expect(token).toBeDefined();
    expect(typeof token).toBe('string');
    expect(token.length).toBeGreaterThan(0);
  });

  test('should call api correctly', async () => {
    const response = await client.request('GET', '/drive/v1/files', {
      params: { page_size: 10 },
    });
    expect(response).toBeDefined();
  });
});
```

#### 运行集成测试
```bash
# 设置测试环境变量
export TEST_FEISHU_APP_ID=xxx
export TEST_FEISHU_APP_SECRET=xxx
export TEST_CLAUDE_API_KEY=xxx

# 运行集成测试
npm run test:integration
```

### 2.3 端到端测试（E2E）
**测试范围**：完整的业务流程测试
**测试场景**：
1. 配置管理流程测试
2. 数据采集流程测试
3. 周报生成流程测试
4. 推送流程测试
5. 定时任务流程测试

#### E2E测试用例示例
```bash
# 测试配置创建
smartflow config create test-team "测试团队"
smartflow config list | grep "test-team"

# 测试数据采集
smartflow generate collect --team test-team --range thisweek --output test-data.json
[ -f test-data.json ] && echo "数据采集成功" || echo "数据采集失败"

# 测试周报生成
smartflow generate run --team test-team --range thisweek --output test-report.json
[ -f test-report.json ] && echo "周报生成成功" || echo "周报生成失败"

# 测试推送功能
smartflow push test test-team
```

#### 自动化E2E测试
可以使用shell脚本或自动化测试框架编写E2E测试：
```bash
#!/bin/bash
set -e

echo "=== 开始E2E测试 ==="

# 1. 测试健康检查
echo "1. 测试健康检查..."
smartflow health
if [ $? -ne 0 ]; then
  echo "健康检查失败"
  exit 1
fi

# 2. 测试配置创建
echo "2. 测试配置创建..."
smartflow config create e2e-test "E2E测试团队"
if [ $? -ne 0 ]; then
  echo "配置创建失败"
  exit 1
fi

# 3. 测试数据采集（mock数据）
echo "3. 测试数据采集..."
# 这里可以使用mock或者测试环境的真实数据

# 清理测试数据
smartflow config delete e2e-test

echo "=== E2E测试全部通过 ==="
```

## 三、功能测试用例设计
### 3.1 配置管理模块测试用例
| 测试用例 | 测试步骤 | 预期结果 |
|---------|---------|---------|
| 创建团队配置 | 执行 `smartflow config create test "测试团队"` | 配置文件创建成功，配置内容正确 |
| 列出团队配置 | 执行 `smartflow config list` | 显示所有已创建的团队配置 |
| 查看团队配置 | 执行 `smartflow config show test` | 显示指定团队的完整配置内容 |
| 删除团队配置 | 执行 `smartflow config delete test` | 配置文件被删除，列表中不再显示 |
| 无效配置校验 | 保存缺少必填字段的配置 | 系统提示配置校验失败，不保存 |

### 3.2 数据采集模块测试用例
| 测试用例 | 测试步骤 | 预期结果 |
|---------|---------|---------|
| 文档采集测试 | 配置文档采集，时间范围内有修改的文档 | 正确采集到所有符合条件的文档，包含标题、修改人、内容摘要 |
| 任务采集测试 | 配置任务采集，时间范围内有状态变更的任务 | 正确采集到所有符合条件的任务，包含标题、状态、负责人 |
| 会议采集测试 | 配置会议采集，时间范围内有会议 | 正确采集到所有符合条件的会议，包含主题、纪要、Action Items |
| 时间范围过滤测试 | 配置时间范围外的数据 | 不会被采集到 |
| 排除规则测试 | 配置排除的关键词或用户 | 符合排除规则的内容不会被采集 |
| 数据源未启用测试 | 关闭某个数据源开关 | 该数据源的数据不会被采集 |

### 3.3 内容生成模块测试用例
| 测试用例 | 测试步骤 | 预期结果 |
|---------|---------|---------|
| 正常生成测试 | 有采集数据的情况下生成周报 | 生成完整的周报内容，包含各个板块 |
| 空数据生成测试 | 没有采集到任何数据 | 提示"本周暂无相关数据"，不报错 |
| 幻觉校验测试 | 生成内容包含不存在的信息 | 校验模块发现并警告，或过滤无效内容 |
| 不同详细程度测试 | 分别配置low/medium/high详细程度 | 内容的详细程度符合配置 |
| 风险识别测试 | 数据中包含延期任务、未跟进决议 | 正确识别并标记对应风险等级 |
| 来源链接测试 | 所有生成的内容项 | 都有对应的来源链接，链接有效 |

### 3.4 推送模块测试用例
| 测试用例 | 测试步骤 | 预期结果 |
|---------|---------|---------|
| 测试推送测试 | 执行 `smartflow push test <teamId>` | 收到测试卡片，推送成功 |
| 群聊推送测试 | 配置群聊渠道，推送周报 | 群聊中收到正确的周报卡片 |
| 用户推送测试 | 配置用户渠道，推送周报 | 用户收到正确的周报卡片 |
| 审核流程测试 | 开启需要审核配置 | 周报先推送给审核人，审核通过后才推送到渠道 |
| 卡片链接测试 | 点击卡片中的来源链接 | 正确跳转到对应的飞书文档/任务/会议 |
| 推送失败处理测试 | 配置无效的渠道ID | 系统提示推送失败，记录错误日志 |

### 3.5 定时任务模块测试用例
| 测试用例 | 测试步骤 | 预期结果 |
|---------|---------|---------|
| 任务添加测试 | 执行 `smartflow schedule add <teamId>` | 定时任务添加成功，列表中显示 |
| 任务触发测试 | 执行 `smartflow schedule trigger <teamId>` | 任务立即执行，生成并推送周报 |
| 任务删除测试 | 执行 `smartflow schedule remove <teamId>` | 定时任务被删除，不再执行 |
| 定时触发测试 | 配置每分钟执行的cron | 每分钟都会触发任务执行 |
| 服务重启测试 | 重启定时任务服务 | 原有任务自动恢复，继续执行 |
| 错误重试测试 | 执行过程中出现错误 | 自动重试最多3次，失败推送告警 |

## 四、性能测试
### 4.1 性能指标
| 指标 | 要求 |
|-----|------|
| 单团队数据采集时间 | ≤ 2分钟 |
| 单团队周报生成时间 | ≤ 1分钟 |
| 同时支持团队数 | ≥ 50个 |
| 定时任务调度误差 | ≤ 1分钟 |
| 系统资源占用 | CPU ≤ 2核，内存 ≤ 4G |

### 4.2 测试方法
```bash
# 压测数据采集
time smartflow generate collect --team large-team --range lastweek

# 压测周报生成
time smartflow generate run --team large-team --range lastweek

# 多团队并发测试
for i in {1..10}; do
  smartflow generate run --team team$i --range lastweek &
done
wait
```

### 4.3 性能优化测试
- 测试不同并发数下的系统表现
- 测试大数据量下的处理能力
- 测试内存泄漏情况（长时间运行监控内存占用）

## 五、验收测试标准
### 5.1 功能验收标准
1. 所有核心功能测试用例通过率100%
2. 飞书API调用成功率≥99%
3. 周报内容准确率≥95%（人工抽查验证）
4. 推送成功率≥99%
5. 定时任务触发准确率≥99%

### 5.2 非功能验收标准
1. 系统连续运行7天无崩溃
2. 错误日志中没有未处理的异常
3. 响应时间满足性能指标要求
4. 所有配置项都有对应的文档说明
5. 错误提示清晰明了，便于问题定位
